# Feature Specification: F-04 — Персонализированная подборка

**Feature Branch**: `004-patient-selection`
**Created**: 2026-05-08
**Status**: Draft
**Dependencies**: F-01 ✅, F-02 ✅, F-03 ✅

## Overview

Координатор клиники формирует персональную подборку эмбрионов для конкретного пациента и передаёт уникальную ссылку. Пациент по этой ссылке видит только те эмбрионы, которые для него отобрал координатор. Координатор отслеживает, открыл ли пациент свою подборку.

Функции данной фичи — завершить рабочий процесс координатора (управление списком пациентов и их подборками), добавить трекинг статуса «открыта ли подборка», а также обеспечить сквозной путь от создания подборки до просмотра пациентом — на уровне API.

## Контекст

Инфраструктура для хранения подборок и токенов уже реализована в F-02:
- Сущности `patients` и `patient_selections` существуют
- Токены выпускаются и отзываются через `POST/DELETE /patients/:id/token`
- Пациент с токеном видит только свои эмбрионы (`GET /embryos` фильтруется по `embryo_ids`)
- Аудит-лог токенов фиксирует каждое использование

F-04 добавляет:
1. API для координатора: список пациентов + расширенный просмотр состояния подборки
2. Трекинг: фиксация момента первого просмотра подборки пациентом (`opened_at`)
3. Сквозной тест полного пользовательского пути

## User Scenarios & Testing

### User Story 1 — Координатор видит список своих пациентов (Priority: P1)

Координатор хочет видеть всех пациентов своей клиники одним списком: у кого есть подборка, у кого её ещё нет, кто уже открыл ссылку.

**Why this priority**: Без этого экрана координатор не может работать с несколькими пациентами — у него нет обзора. Это точка входа во весь рабочий процесс.

**Independent Test**: `GET /api/v1/patients` с токеном координатора возвращает массив пациентов его клиники. Каждый элемент содержит признак наличия подборки и статус просмотра.

**Acceptance Scenarios**:

1. **Given** координатор клиники A создал двух пациентов и одному добавил подборку, **When** выполняется `GET /api/v1/patients`, **Then** возвращается массив из 2 записей; у одного `selection: { opened_at: null, token_expires_at: "..." }`, у другого `selection: null`
2. **Given** у пациента есть подборка, но токен ещё не использовался, **When** координатор запрашивает список, **Then** `selection.opened_at: null`
3. **Given** пациент открыл подборку, **When** координатор запрашивает список, **Then** `selection.opened_at` содержит временную метку первого открытия
4. **Given** координатор клиники B, **When** выполняется `GET /api/v1/patients`, **Then** пациенты клиники A не попадают в ответ
5. **Given** запрос без авторизации, **When** `GET /api/v1/patients`, **Then** 401
6. **Given** admin без параметра `?clinic_id=`, **When** `GET /api/v1/patients`, **Then** 400
7. **Given** admin с параметром `?clinic_id=clinic-A`, **When** `GET /api/v1/patients`, **Then** список пациентов клиники A

---

### User Story 2 — Координатор видит детальное состояние подборки пациента (Priority: P2)

Координатор открывает карточку конкретного пациента и видит полную картину: какие эмбрионы включены в подборку, до какого числа действует ссылка, открывал ли пациент её.

**Why this priority**: Нужен для управления подборкой: обновления состава, продления срока, понимания вовлечённости пациента.

**Independent Test**: `GET /api/v1/patients/:id` возвращает данные пациента + состояние подборки (embryo_ids, token_expires_at, opened_at).

**Acceptance Scenarios**:

1. **Given** у пациента есть подборка из 3 эмбрионов и действующий токен, **When** `GET /api/v1/patients/:id`, **Then** ответ содержит `selection.embryo_ids` (3 UUID), `selection.token_expires_at` (дата), `selection.opened_at: null`
2. **Given** пациент ни разу не использовал токен, **When** координатор запрашивает карточку, **Then** `selection.opened_at: null`
3. **Given** у пациента нет подборки, **When** `GET /api/v1/patients/:id`, **Then** `selection: null`
4. **Given** запрос к пациенту другой клиники, **When** координатор делает `GET /api/v1/patients/:id`, **Then** 404
5. **Given** несуществующий id, **When** `GET /api/v1/patients/:id`, **Then** 404

---

### User Story 3 — Фиксация первого открытия подборки пациентом (Priority: P2)

Когда пациент впервые использует свой токен для просмотра эмбрионов, система автоматически запоминает дату и время этого события. Координатор затем видит это через `GET /patients/:id`.

**Why this priority**: Ключевой индикатор вовлечённости — координатор знает, дошла ли ссылка до пациента и открыл ли он её.

**Independent Test**: Полный сквозной путь: создать пациента → добавить эмбрионы → выдать токен → использовать токен → проверить `opened_at` в `GET /patients/:id`.

**Acceptance Scenarios**:

1. **Given** пациент получил токен и подборка ранее не открывалась, **When** пациент выполняет `GET /api/v1/embryos` с токеном, **Then** поле `opened_at` в `patient_selections` устанавливается в текущее время
2. **Given** пациент уже открывал подборку (`opened_at` установлено), **When** пациент снова выполняет `GET /api/v1/embryos`, **Then** `opened_at` не перезаписывается — сохраняется время первого открытия
3. **Given** пациент использовал токен, **When** координатор запрашивает `GET /api/v1/patients/:id`, **Then** `selection.opened_at` равно времени первого открытия
4. **Given** токен истёк, **When** пациент пытается его использовать, **Then** 401, `opened_at` не изменяется

---

### Edge Cases

- Координатор запрашивает список когда у него 0 пациентов → пустой массив `[]`, не ошибка
- Пациент существует, но у него нет подборки → `GET /patients/:id` возвращает `selection: null` без ошибки
- Токен отозван координатором → пациент получает 401, `opened_at` не изменяется
- Координатор обновляет `embryo_ids` после того, как пациент уже открыл подборку → `opened_at` не сбрасывается; пациент при следующем запросе видит актуальный состав
- Одновременные запросы от пациента (race condition на `opened_at`) → `opened_at` устанавливается один раз благодаря условному UPDATE (`SET opened_at = NOW() WHERE opened_at IS NULL`)
- Пациент с истёкшим токеном: попытка продлить срок через `POST /patients/:id/token` с новым `ttl_days` — корректно заменяет токен, `opened_at` не сбрасывается
- Admin без `?clinic_id=` → 400, с `?clinic_id=` → пациенты только той клиники; `GET /patients/:id` для admin не требует параметра (id однозначно идентифицирует пациента)

## Requirements

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять координатору полный список пациентов его клиники через защищённый эндпоинт; каждый элемент списка содержит вложенный объект `selection` (или `null`) с полями `opened_at` и `token_expires_at`
- **FR-002**: Система ДОЛЖНА предоставлять координатору детальную карточку пациента, включающую: идентификаторы эмбрионов в подборке, срок действия активного токена (если есть), время первого открытия подборки пациентом (если открывалась)
- **FR-003**: Доступ к данным пациентов ДОЛЖЕН быть ограничен клиникой: координатор видит только пациентов своей клиники; запрос к пациенту другой клиники возвращает 404
- **FR-004**: При первом успешном использовании токена пациентом система ДОЛЖНА зафиксировать временную метку в записи подборки; при повторных обращениях метка НЕ перезаписывается
- **FR-005**: Все операции чтения данных пациентов ДОЛЖНЫ быть доступны только для ролей `coordinator` и `admin`; неавторизованный запрос возвращает 401
- **FR-006**: Администратор ДОЛЖЕН иметь возможность просматривать пациентов любой клиники через те же эндпоинты, передав обязательный параметр `?clinic_id=`; запрос без `clinic_id` от admin возвращает 400
- **FR-007**: Список пациентов ДОЛЖЕН быть отсортирован по дате создания в порядке убывания (новые сначала)

### Key Entities

- **Patient** (существует из F-02): пациент клиники; атрибуты: id, name, clinic_id, created_by, created_at
- **PatientSelection** (существует из F-02, расширяется): подборка эмбрионов для пациента; атрибуты: id, patient_id, embryo_ids, created_by, created_at, updated_at; **новый атрибут**: `opened_at` (время первого просмотра пациентом, null до первого открытия)
- **AccessToken** (существует из F-02): токен доступа пациента; атрибуты: token_value, patient_id, selection_id, expires_at, revoked_at; используется для определения `token_expires_at` в ответе координатора

## Success Criteria

### Measurable Outcomes

- **SC-001**: Координатор получает список своих пациентов без ожидания — ответ приходит мгновенно при любом количестве пациентов в клинике (до нескольких сотен)
- **SC-002**: После первого открытия подборки пациентом координатор видит `opened_at` при следующем обращении к карточке пациента — без дополнительных действий
- **SC-003**: Данные пациентов одной клиники недоступны сотрудникам другой клиники — проверяется автоматическими тестами на изоляцию
- **SC-004**: Сквозной сценарий «создать пациента → добавить эмбрионы → выдать токен → пациент открывает → координатор видит opened_at» выполняется корректно и покрыт интеграционным тестом
- **SC-005**: Все новые эндпоинты (`GET /patients`, `GET /patients/:id`) защищены авторизацией — проверяется тестом покрытия маршрутов

## Assumptions

- Создание пациентов и управление подборками (добавление эмбрионов, выдача токенов) уже реализованы в F-02 и не переделываются
- Пагинация списка пациентов не требуется на уровне MVP: клиника работает с десятками, максимум сотнями пациентов — полный список в одном ответе допустим
- Email или мессенджер-отправка ссылки пациенту — вне scope; координатор копирует ссылку вручную
- UI координатора реализуется отдельно в F-10; F-04 поставляет только API
- `opened_at` фиксируется при первом успешном вызове `GET /api/v1/embryos` с валидным токеном; другие эндпоинты (например, `GET /embryos/:id`) не влияют на `opened_at` в рамках F-04
- Если у пациента несколько токенов (новый выдан взамен старого), `opened_at` хранится на уровне подборки и не сбрасывается при перевыдаче токена
- Фильтрация и поиск в списке пациентов — вне scope F-04 (реализуется в F-10 на уровне UI)

## Clarifications

### Session 2026-05-08

- Q: Нужна ли пагинация для `GET /api/v1/patients`? → A: Нет, полный список для MVP (клиника — десятки пациентов)
- Q: Какой эндпоинт триггерит `opened_at`? → A: Только `GET /api/v1/embryos` (список подборки); отдельные карточки `GET /embryos/:id` не считаются
- Q: Структура ответа списка пациентов — вложенный объект или плоские поля? → A: Вложенный объект `selection: { opened_at, token_expires_at } | null` — одинаково в списке и в детальной карточке
- Q: Поведение `GET /patients` для admin — все клиники или обязательный `?clinic_id=`? → A: `?clinic_id=` обязателен для admin; без него 400; `GET /patients/:id` для admin без параметра (id однозначен)
