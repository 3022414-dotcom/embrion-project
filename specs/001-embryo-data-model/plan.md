# Implementation Plan: F-01 — Embryo Data Model

**Branch**: `001-embryo-data-model` | **Date**: 2026-05-01 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-embryo-data-model/spec.md`

## Summary

Define the embryo data schema as a shared TypeScript package (`packages/schema`) within a
backend/frontend monorepo. The package delivers: (1) strict TypeScript types for all embryo
fields, (2) Zod runtime validators enforced at the storage boundary, (3) a role-based
visibility projection layer that strips restricted fields per caller role, and (4) a
semver-versioned JSON schema manifest. Both `apps/api` (Fastify) and `apps/web` (React)
consume this package — zero duplication of type definitions across the stack.

## Technical Context

**Language/Version**: TypeScript 5.4 (strict mode) across the full monorepo
**Primary Dependencies**: Zod 3.x (runtime schema + validation), pnpm workspaces +
Turborepo (monorepo tooling), Fastify 4.x (backend), React 18 + Vite 5 (frontend),
PostgreSQL 16 (storage), Vitest 1.x (testing), @testcontainers/postgresql (integration tests)
**Storage**: PostgreSQL 16 — normalized relational columns (not JSONB); sub-records map to
typed columns; enables indexed queries on donor traits and status
**Testing**: Vitest (unit + integration), Supertest (HTTP integration), real PostgreSQL via
testcontainers (no mocks for storage layer per Constitution Principle II)
**Target Platform**: Linux server (Node.js 20 LTS) + modern web browser (ES2022+)
**Project Type**: Web application — monorepo (backend API + frontend SPA)
**Performance Goals**: Single embryo fetch p95 < 200ms; Zod schema validation < 1ms per
record; role projection < 0.1ms
**Constraints**: No raw PII in server logs; role visibility enforced exclusively at the
data layer (`packages/schema`), not in UI components or HTTP handlers
**Scale/Scope**: Clinic-scale — ~1,000–10,000 embryo records; single clinic deployment
initially; schema designed to be forward-compatible with multi-clinic in future

**Structure Decision**: Monorepo (web application layout) with a dedicated
`packages/schema` workspace as the canonical owner of embryo types, Zod validators, and
role visibility rules. `apps/api` and `apps/web` both depend on `packages/schema` — this
guarantees a single source of truth for the data model across the stack and satisfies
Constitution Principle I (single responsibility, no duplication).
